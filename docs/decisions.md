# Decisions

Decisions made during planning and execution, with the reasoning behind each.

## Decisions taken

| # | Decision | Choice | Reason |
|---|---|---|---|
| 1 | Number of machines | **4** | Matches the 4 available 1G ports exactly; a 4th machine adds real value (3 OSD hosts + multiple compute nodes) over a 3-machine layout. |
| 2 | Machine to retire | **5080** (dead PSU) | Originally the 7050 was to be retired (weakest CPU), but the 5080's power supply failed and the 7050's PSU is incompatible, so the **5080** is retired as the donor instead and the **7050 returns** as the fourth node. The 7050's weak CPU is acceptable because it serves mainly as an I/O-bound OSD/compute node. |
| 3 | Controller | **7071** | 8 cores, mini-tower chassis, the most RAM (32GB), and now NVMe-booted. The natural "brain" of the cluster. |
| 4 | RAM split | **32 / 16 / 16 / 16** GB | The controller is the most service-dense node (control plane + Ceph MON/MGR), so it gets 32GB. Uses all 10 sticks (4+2+2+2), now across 7071/7060/5090/7050. |
| 5 | Ceph OSD layout | **5 OSDs across 3 hosts (2 + 1 + 2)** | Minimum 3 OSD hosts for default 3Ă— replication with a `host` failure domain. 7060 carries 2 OSDs, 5090 carries 1, 7050 carries 2. Better balanced than the original 3+1+1, and all OSDs are matched 250GB SATA SSDs. (Per-tower SATA capacity is 3 *total* = 1 boot + 2 OSD, which caps each tower at 2 OSDs.) |
| 6 | Network speed | **1G, flat** | 10/100 is too slow for Ceph replication; a single flat network is fine for a learning cluster. Ceph saturating the 1G link is an intended, instructive observable. |
| 7 | Deployment method | **Phase 0 (prep/OS) â†’ Phase 1 (manual Ceph + controller) â†’ Phase 2 (hand-rolled Ansible compute) â†’ Phase 3 (Kolla-Ansible rebuild)** | Explicit phased learning progression; each phase is motivated by the friction of the previous one. |
| 8 | Cluster lifespan | **Temporary (a few days)** | Build, benchmark, tear down. This is a test/learning cluster, not a permanent deployment. |
| 9 | Operating system | **AlmaLinux 9**, Minimal Install, SELinux enforcing | Switched from an initial lean toward AlmaLinux 10: RDO's EL10 packaging was still maturing in early 2026, which would bite the manual Phase 1 install hard. EL9 has mature, well-trodden RDO repos. |
| 10 | OpenStack release | **2025.1 "Epoxy"** | Recent release with complete, stable RDO/EL9 repos and lots of install-guide coverage â€” the sweet spot on EL9. |
| 11 | Ceph version | **Squid (19.x)** (Reef 18.x acceptable fallback) | Pairs with Epoxy-era OpenStack; installed via `centos-release-ceph-squid` on EL9. |
| 12 | Ceph bootstrap method | **cephadm** | Chosen by the user over the assistant's lean toward a fully-manual MON/MGR/OSD bring-up. Trade-off on record: cephadm is itself a container-orchestration layer, so Kolla in Phase 3 becomes the *second* such layer, not the first. Learning value holds because daemons are still placed by hand. |
| 13 | IP addressing | **Static IPs, no DHCP, no reservations** | Deterministic, always-known addressing (Ceph MONs configured by IP, Keystone endpoints are URLs, Kolla inventory is IP-based). Avoids dependence on the home router's lease table and guarantees no interference with the home network. Confirmed on subnet `192.168.1.0/24`: `controller` .130 (7071), `compute1` .131 (7060), `compute2` .132 (5090), `compute3` .133 (7050). (Name resolution and gateway settled separately â€” see #21.) Full address plan in [inventory.md](inventory.md). |
| 14 | Tenant networking | **VXLAN self-service** (Open vSwitch driver â€” see #24) | Tenant networks tunnel VM traffic over the `192.168.1.0/24` underlay, sealing VM DHCP inside the overlay so it cannot race the home router's DHCP server (the "dual-DHCP" problem). A Neutron router NATs to a **flat provider/external network** on `192.168.1.0/24`; VMs reach the outside via **floating IPs**. Achieves the isolation a VLAN would give but in software, with no managed switch. (Supersedes the earlier "overlays + provider-on-its-own-VLAN" sketch.) |
| 15 | Ceph MON quorum | **4 MONs (controller + compute1/2/3), as auto-placed by cephadm** | Originally recorded as a deliberate **single** MON on the controller (accepted SPOF). In practice cephadm's default MON placement deploys a MON on each host as it is enrolled, so the live cluster actually runs **4 MONs** â€” confirmed by `ceph -s` (`1/4 mons down, quorum controller,compute1,compute2` during the compute3 outage). Resilience-positive: the intended single-MON SPOF never actually existed, so this is left as-is. Learning note: 4 is an *even* count â€” quorum is 3, so it tolerates only **one** MON failure (same as 3 would) and a 2/2 partition has no quorum; an odd count (3 or 5) is strictly better. Not worth pruning on a throwaway cluster, and Phase 3's Kolla sets its own MON count. |
| 16 | OSD memory target | **Lowered to ~1.5â€“2GB** (from the 4GB default) | The 16GB OSD nodes cannot afford 4GB BlueStore cache per OSD; baked into the Ceph config from the start. |
| 17 | Domain / DNS suffix | **`lab.internal`** | Confirmed by the user (FQDNs `controller.lab.internal`, `compute1â€“3.lab.internal`). Supersedes the earlier *proposed* `cluster.lab.internal`. A reserved/non-routable TLD that cannot collide with a real domain. |
| 18 | NVMe placement | **Both NVMe used as boot disks: 7071 (harvested from 5080) + 5090 (its own)** | The controller is the one node where root-disk latency materially helps (MariaDB, RabbitMQ, Keystone token churn touched by every benchmarked API call). Keeping the NVMe out of the OSD pool keeps all OSDs on matched SATA SSDs, avoiding a heterogeneous pool that would muddy benchmarks. |
| 19 | Per-tower SATA capacity | **3 SATA SSDs total per tower = 1 boot + 2 OSD** | The 7060 and 7050 each have 3 SATA bays/cables *total*, not 3 OSDs plus a boot disk. This caps each tower at 2 OSDs and drives the 2+1+2 topology. (Corrects an earlier over-count in the chunk-01 build plan.) |
| 20 | Hyperthreading | **Left enabled** on the compute nodes (7060, 5090, 7050) | RAM, not CPU, is the binding constraint on the 16GB nodes, so more logical CPUs is the more useful default; it is also the BIOS default (one less variable across four machines). The 7071 (i7-9700) has no HT. Caveat for benchmarking: HT adds variance to CPU-bound results â€” report vCPU counts as physical cores and interpret accordingly rather than toggling BIOS mid-project. |
| 21 | Name resolution | **Local `/etc/hosts` on all nodes, no DNS** | A 4-node throwaway cluster doesn't need real DNS; identical `/etc/hosts` files map all four FQDNs/short names. Nodes still use the normal LAN gateway for outbound internet (package/container pulls). |
| 22 | Firewall | **firewalld disabled** | Pragmatic for a short-lived, isolated lab â€” removes a whole category of "why can't this daemon talk" debugging. Keeping it on would mean opening a long per-service port list with little Phase 1 learning value. |
| 23 | Ceph hostnames | **FQDNs (`*.lab.internal`) with `cephadm bootstrap --allow-fqdn-hostname`** | Keeps Ceph identifiers consistent with the OS hostnames and `/etc/hosts`; the bootstrap check guards against mixing short and FQDN names, not against FQDNs. All `ceph orch host add` use FQDNs. Cost is purely cosmetic (longer daemon/host names). |
| 24 | Neutron mechanism driver | **Open vSwitch (OVS)** â€” *was Linux bridge* | Linux bridge was the original pick (fewest moving parts for a learning cluster), but **RDO 2025.1 "Epoxy" ships no linuxbridge agent** â€” the `openstack-neutron*` set has only OVS / OVN / macvtap / sriov agents, and `dnf provides '*/neutron-linuxbridge-agent'` finds the binary only *inside a kolla container image*, not an installable RPM (linuxbridge has been deprecated for years; the upstream 2025.1 install guide moved to OVN for the same reason). OVS is the minimal pivot: it preserves the VXLAN self-service model (#14) and the entire agent-based Stage 3â€“6 plan (ML2 + separate L3/DHCP/metadata agents, VXLAN tenant networks, MTU 1450), changing only the L2 agent and its config (`openstack-neutron-openvswitch`, `openvswitch_agent.ini` with `tunnel_types = vxlan`, and an OVS provider bridge for the flat external net). OVN considered and rejected for Phase 2 â€” see R12. |
| 25 | Network node | **The controller (7071)** | The 7071 runs the Neutron L3, DHCP, and metadata agents in addition to its control-plane and Ceph MON/MGR roles. Concentrating the network node on the controller fits the single-NIC, days-long cluster. |
| 26 | Ansible scope | **Only the repeated compute-node work is rolled into Ansible** (`nova_compute`/`neutron_compute`, plus a throwaway `common` role for learning); controller-side Nova/Neutron is done **manually** (one-time) | The Ansible "seam" is repetition, and the one-time controller bring-up isn't repeated â€” forcing it into a role would add complexity without the payoff. Compute1/2/3 are the "same steps three times," which is where roles + per-host inventory variables earn their place. Also serves the learning goal: build understanding on a low-stakes `common` role first. |
| 27 | Ansible distribution | **Installed via `uv` (`uv tool install ansible --python 3.12`) â€” community 13.7 / `ansible-core` 2.20 on Python 3.12** | The AlmaLinux `dnf` package ships an end-of-life `ansible-core` (2.14/2.15) too old to match the live docs site (oldest published is community 11 / core 2.18). `uv` keeps it isolated from the system Python; pinning Python 3.12 was required because AlmaLinux's system Python 3.9 silently capped the resolver at `ansible` 8.7.0. Community 13 matches the docs "13" link exactly. The full community package was chosen (over bare `ansible-core`) so `openstack.cloud` is bundled for Stage 5. |
| 28 | Ansible privilege model | **Run as the normal login user; per-task `become` (default-off in `ansible.cfg`); `sudo` left password-protected (not passwordless); `-K` per escalating run** | Never run Ansible as root or via `sudo ansible-playbook` (that would SSH rootâ†’root and read root's keys/config). Default-off escalation keeps the privilege surface visible in the roles and makes a forgotten `become:` fail loudly rather than silently run as root (the inverse of the Phase 1 issue #3 root-owned-file class). Password-protected sudo is the deliberate security choice over passwordless; `-K` keeps the prompt explicit per run. |
| 29 | Per-host address variable | **One per-host `local_ip` (defined on all four nodes) serves as both the underlay IP and the VXLAN tunnel endpoint â€” no separate `underlay_ip`** | With one NIC per node, the host's `192.168.1.x` underlay address and its VTEP address are always the same value, so a second variable would be pure redundancy. `local_ip` is defined on the controller too (`.130`), since the controller is also a VTEP (it runs the L3/DHCP agents on tenant networks). The same variable is reused for inventory-driven `/etc/hosts` rendering. |
| 30 | cephadm SSH access | **Restore root SSH for cephadm, hardened: key-only (`PermitRootLogin prohibit-password`) + `from=`-restricted to the cluster nodes (`192.168.1.128/29`)** | cephadm manages every host over its **own** SSH channel as `root`, with a cluster keypair it generates â€” a separate control plane from Ansible. Root's authorized key had been pruned on the assumption that Ansible's login-user + `become` model (#28) made root SSH unnecessary; true for Ansible, but it broke cephadm (`CEPHADM_HOST_CHECK_FAILED` / ssh auth failure for root). cephadm needs root-level command execution, and the only two ways to give it that are root SSH or a non-root user with **passwordless** sudo (it is fully automated, so no `-K`-style prompt is possible). Chose to restore the root key â€” hardened to key-only via an sshd drop-in and `from=`-restricted so it only works from a cluster node (covers an mgr failover to any host; `192.168.1.128/29` spans `.128â€“.135`, i.e. all four nodes) â€” rather than add a NOPASSWD service user (see R10). Deliberately distinct from #28's Ansible login user: the two management planes keep separate, intentional privilege postures. |
| 31 | Nova ephemeral disk backend | **Ceph RBD-backed** (`vms` pool, `client.nova` auth, per-compute libvirt secret, `[libvirt] images_type = rbd`) | Closes the Phase 2 open item (local qcow2 vs. RBD). RBD-backed ephemeral boots VMs from Ceph: it enables live migration, lets Ceph recovery visibly affect running VMs (an instructive benchmark observable), reuses the Phase 1 keyring-permissions lesson (issue #3), and **shares the per-compute libvirt Ceph secret that Cinder (#32) also needs** â€” so adding Cinder afterward is nearly free on the compute side. Cost: VM root-disk I/O now rides the 1G underlay alongside Ceph/management/VXLAN traffic, itself worth measuring. Provision the `vms` pool + `client.nova` before templating the Stage 4 libvirt config. |
| 32 | Add Cinder (block storage) | **Yes â€” as Phase 2 Stage 6, RBD-backed** (`volumes` pool, `client.cinder`; controller-side `cinder-api`/`-scheduler`/`-volume`) | Gives the deferred Phase 1 `volumes` pool a home. Placed at the **end** of Phase 2, after a VM boots on RBD ephemeral (Stage 5), so it stays a self-contained block-storage module that doesn't bloat the critical path or overload the already-heavy Stage 3. Controller-side install reuses the Phase 1 Glance service-account + RBD-keyring pattern; the compute side needs only the libvirt Ceph secret #31 already provisions in Stage 4. `cinder-volume` runs on the controller (the 32GB node) â€” no new per-compute daemons, so RAM is not a constraint. Alternative considered: folding the controller-side services into Stage 3 â€” rejected to avoid overloading the Nova-cells + VXLAN bring-up. Cinder is genuinely optional on a throwaway benchmark cluster (Glance-on-RBD already exercises Ceph integration); chosen here for the block-storage learning and volume-I/O benchmarking. |
| 33 | RabbitMQ / Erlang version | **Stay on the RDO-native RabbitMQ 3.9 (SIG) track; pin Erlang to the SIG's 24.x by excluding `erlang*` from EPEL** | First real use of RabbitMQ (Nova as its consumer) exposed it running rabbit **3.9.21** on **Erlang 26.2.5** â€” an unsupported pairing (3.9 tops out at Erlang 24) that accepted then reset every AMQP connection. RDO pulls rabbit from the CentOS Messaging SIG (`centos-rabbitmq-38`), which ships a *matched* Erlang 24 â€” but **EPEL also ships Erlang 26 and dnf picked it on version number alone**, silently installing an incompatible Erlang under rabbit 3.9. Fix: reinstall with `--disablerepo=epel` (gets `rabbitmq-server 3.9.21` + `erlang 24.3.4.2` from the SIG), made durable with `dnf config-manager --setopt=epel.excludepkgs=erlang* --save`. Kept the RDO-intended versions rather than chasing rabbit 4.x or containerizing (see R11). The separate epmd-before-network boot-ordering fix is logged in [project-phase-2-stage-3.md](project-phase-2-stage-3.md). **Lesson on record: EPEL can outrank a SIG package by version and break an RDO dependency â€” exclude it.** |
| 34 | Neutron OVS agent SELinux fix | **Enable the `os_neutron_dac_override` boolean** (`setsebool -P`) â€” *not* a custom policy module or `permissive` | The OVS agent's privsep helper (root, confined `neutron_t`) was denied `CAP_DAC_OVERRIDE` and crashed at startup. `audit2allow` revealed `openstack-selinux` already ships the needed `allow` behind this **default-off boolean**, so flipping it is the vendor-intended fix: nothing custom to maintain, survives policy updates, stays `Enforcing`. A capability can't be file-scoped (it's all-or-nothing in the kernel), so this is the narrowest *correct* form; `dac_override` is modest beside the `net_admin`/`sys_admin` `neutron_t` already holds, and the boolean also covers the L3/DHCP agents (which use privsep lazily). The one remaining **one-shot `neutron_t`â†’`cache_home_t` `create` denial** (root helper trying to write `/root/.cache`) is left **denied on purpose** â€” allowing it would be *broader* than the status quo (Neutron writing root's home), nothing functional needs it (the agent caches fine in `neutron_var_lib_t`), and it's non-fatal. Full log in [project-phase-2-stage-3.md](project-phase-2-stage-3.md). |
| 35 | Keeping `/etc/ceph/ceph.conf` glance-readable under SELinux | **`restorecond` watches the file and restores its label on every cephadm rewrite** | cephadm maintains `/etc/ceph/ceph.conf` on the `_admin` controller and rewrites it via tempfile-then-`os.rename()`, which preserves a `user_tmp_t` label that confined `glance_api_t` can't read ([Ceph #9530](https://tracker.ceph.com/issues/9530)) â€” so glance-api dies on `conf_read_file` after each rewrite, and a one-off `restorecon` doesn't hold. Chose **`restorecond`** (add `/etc/ceph/ceph.conf` to `/etc/selinux/restorecond.conf`, enable `restorecond.service`): the OS-native daemon built for "files recreated with the wrong label," and the same relabel-after-write approach Ceph upstream uses for this class ([ceph PR #23278](https://github.com/ceph/ceph/pull/23278)). Rejected: a hand-rolled systemd `path`+`restorecon` unit (functionally identical, but reinvents `restorecond`); **decoupling** glance with a static minimal `ceph.conf` + service-only keyring (the standard production pattern â€” but extra divergence on a converged Ceph-and-OpenStack node, which also legitimately needs the admin `ceph.conf` for the CLI); and **dropping the `_admin` label** (stops the rewrites but loses cephadm's auto-maintenance of the `client.admin` keyring/CLI on the controller). Log in [project-phase-2-stage-3.md](project-phase-2-stage-3.md) problem 6. |
| 36 | Compute virtualization type | **`kvm` (hardware-accelerated; VT-x confirmed)** | Closes the Phase 2 `kvm`/`qemu` open item. All three compute CPUs expose the `vmx` flag (`grep -c -w vmx /proc/cpuinfo` â†’ 24/32/16 logical CPUs), confirming Intel VT-x is enabled in BIOS, so `[libvirt] virt_type = kvm` runs guests on real hardware virtualization rather than the slow `qemu` software-emulation fallback. The `nova_compute` role asserts `vmx > 0` and **fails loudly per node**, so a BIOS regression (VT-x disabled) can't silently degrade a node to emulation â€” it stops the play instead. |
| 37 | `ansible-vault` layout for compute secrets | **vars/vault split, scoped to the `compute` group** | `group_vars/compute/vault.yml` (encrypted; holds only `vault_*`-prefixed raw secrets) + `group_vars/compute/vars.yml` (committed; maps friendly names â†’ `{{ vault_* }}`). Roles reference only the friendly names, so the indirection stays greppable and the encrypted blob minimal. Run with `--ask-vault-pass` (required for *any* play/ad-hoc touching the `compute` group, since loading its group_vars decrypts the vault). **Five** secrets: the RabbitMQ password and the `nova`/`placement`/`neutron` **Keystone** passwords + the `client.nova` Ceph key. Deliberately **excludes** the nova/neutron **DB** passwords and the `metadata_proxy_shared_secret` â€” compute nodes are DB-less (they reach the DB only via `nova-conductor`) and aren't the metadata node, so those secrets never leave the controller (a real blast-radius reduction). Non-secret cluster facts (controller FQDN, release, Ceph `fsid`/`mon_host`, the `vms` pool name, the libvirt secret UUID) stay plaintext in `group_vars/all.yml` / role `defaults` â€” none is a credential. |
| 38 | `client.nova` key + `ceph.conf` delivery to compute nodes | **Role-placed minimal `ceph.conf` + keyring file (for nova/`rados`) + libvirt secret (for qemu), all from one vaulted key** | Compute nodes have **no `/etc/ceph` by default** (`mgr/cephadm/manage_etc_ceph_ceph_conf = false` governs only the all-hosts push), and RBD-backed Nova (#31) needs Ceph auth via **two** independent paths: `nova-compute` itself (`rados`) reads a keyring file, while **qemu/libvirt** uses a libvirt secret. So the `nova_compute` role places: (a) a **minimal client** `/etc/ceph/ceph.conf` (just `fsid` + `mon_host`, *not* the controller's admin conf); (b) `/etc/ceph/ceph.client.nova.keyring` (`root:nova`, `0640`); and (c) a libvirt secret with a **single shared `rbd_secret_uuid`** across all computes (a UUID is an identifier, not a credential). Both the keyring and the secret value render from the one vaulted `ceph_nova_key`. Ansible's `template` module applies the **policy-correct SELinux context on write**, so no `restorecon` chaser is needed here â€” the opposite of cephadm's `os.rename` mislabel (#35), because Ansible creates the file fresh rather than renaming a temp file. The same key/secret is reused unchanged by Stage 6 Cinder (#31/#32); only its Ceph caps get extended to the `volumes` pool then. |
| 39 | Stage 5 tenant/provider network parameters | **Tenant `10.0.0.0/24` VXLAN @ MTU 1450; provider/external **flat** on `192.168.1.0/24` (DHCP **off**, gateway `192.168.1.1`); floating-IP pool `192.168.1.160â€“.191`** | Closes the three Stage-5 networking open items (tenant CIDR / floating-IP pool / MTU) in one record. **Tenant net:** a `10.0.0.0/24` VXLAN overlay with **MTU 1450** â€” VXLAN adds ~50 bytes of header over the 1500-MTU underlay, so the tenant MTU is lowered rather than the underlay raised (the narrower change), avoiding the classic "SSH connects then hangs" fragmentation. **Provider net:** **flat** on the *real* home `192.168.1.0/24` so the **home router stays the gateway** (`.1`); Neutron runs **no DHCP** on it â€” sealing VM DHCP inside the overlay is the entire point of the VXLAN model (#14, principle 8), and a Neutron DHCP server on the home L2 would re-create the dual-DHCP race that model removes (R8). **Floating-IP pool `.160â€“.191`:** expressed as a `/27`-sized **allocation pool** (`start=192.168.1.160, end=192.168.1.191`) *restricting* the `/24` provider subnet â€” **not** a separate `/27` subnet. Keeping the subnet `/24` is what leaves the home router as gateway and makes `.160`/`.191` ordinary host addresses (not network/broadcast), so all **32** are usable floating IPs; restricting the allocation pool keeps every address Neutron hands out (the router's external-gateway port + floating IPs) inside `.160â€“.191`, away from the home LAN. Overlap **verified against the live router**: home DHCP `.10â€“.49`, router static IPs `.199â€“.225`, and cluster hosts `.130â€“.133` are all outside `.160â€“.191`. History: the pool was pencilled in during the Stage-3 session but **deliberately left unrecorded** pending this DHCP-boundary confirmation (and its provisional "#33" slot was taken by the RabbitMQ/Erlang fix); the confirmation is now in hand, so it is recorded here as **#39**. |

## Decisions deliberately NOT taken (rejected options)

| # | Option considered | Verdict | Reason |
|---|---|---|---|
| R1 | Frankenstein power (110Vâ†’SATA adapters) for a 2nd SATA SSD in the SFFs | **Rejected** | External power bricks dangling from a sealed SFF chassis is a corruption risk that works for a week then quietly fails. A learning cluster does not need balanced OSDs, and the "really good reason" bar isn't met. |
| R2 | Drop the 7060 to free RAM (3-node, 4/4/2 sticks) | **Rejected** | Would cut Ceph to 2 OSD hosts â€” cannot satisfy 3Ă— replication with a `host` failure domain â€” and would lose a 6-core compute node. The RAM gain (one node from 16â†’32GB) doesn't justify crippling the storage layer. |
| R3 | Use the 10/100 switch instead of 1G ports | **Rejected** | Ceph recovery over 100Mbit would be painful and teach bad latency lessons. Stay on 1G. |
| R4 | Kolla-Ansible-only build (skip the manual + hand-rolled Ansible phases) | **Rejected** | Although tempting given the temporary cluster, learning Ansible and OpenStack internals is an explicit goal, so the 3-phase progression was kept. |
| R5 | Drop to a 3-node cluster after the 5080 PSU failure | **Rejected** | A 3-node cluster (7071 + 7060 + 5090) would leave only 2 OSD hosts (7060 + 5090), which cannot satisfy 3Ă— replication with a `host` failure domain. The weak 7050 is worth keeping precisely because it restores a third OSD host. |
| R6 | Put the harvested NVMe in a tower as an OSD (to reach 6 OSDs) | **Rejected** | One NVMe OSD in an otherwise all-SATA pool creates a heterogeneous pool that muddies benchmark numbers, and it would leave the controller booting from SATA. Matched OSDs + a fast NVMe controller is the better setup. |
| R7 | Boot the towers from USB / ODD bay to free all 3 SATA bays as OSDs | **Rejected** | USB boot disks are slow and flaky â€” an unwanted variable in a benchmark cluster. |
| R8 | Flat provider networking for tenants (VMs directly on `192.168.1.0/24`) | **Rejected** | Puts VM DHCP on the same L2 broadcast domain as the home router's DHCP server â€” a dual-DHCP race where either server can answer the other's clients. Disjoint pools prevent duplicate IPs but not the cross-answering. |
| R9 | VLAN-based tenant isolation | **Rejected** | Would cleanly solve the dual-DHCP race, but requires a managed switch that is not in play (the 10/100 switch was set aside). VXLAN achieves the same isolation in software. |
| R10 | Non-root cephadm SSH user with passwordless (NOPASSWD) sudo | **Rejected** | The documented way to avoid root SSH, but the user must have **passwordless** sudo (cephadm is fully automated â€” no prompt), which makes it root-equivalent in blast radius if its key is compromised, and adds a second privileged account to manage. For a throwaway lab the gain is policy/audit, not reduced privilege, so hardened root SSH (#30) was kept instead. |
| R11 | Move RabbitMQ to the 4.x track (repo swap) or run it in a Podman container | **Rejected** | Both were viable escapes from the Erlang-26 mismatch, but unnecessary once the real cause (EPEL shadowing the SIG's Erlang 24) was found. The repo swap is blocked anyway â€” removing `centos-release-rabbitmq-38` cascades to remove `centos-release-openstack-epoxy` (the entire RDO 2025.1 repo set), and installing `centos-release-rabbitmq-4` alongside `-38` hits a file conflict (both own `/etc/yum.repos.d/CentOS-Messaging-rabbitmq.repo`). A Podman rabbit (the box already runs Podman via cephadm) would work and previews Phase 3's containerization, but diverges from the RPM/systemd control plane for no benefit when a one-line EPEL exclude keeps the stack RDO-native (#33). |
| R12 | OVN as the Phase 2 Neutron backend | **Rejected (for Phase 2)** | When linuxbridge turned out to be unpackaged (#24), OVN was the other candidate â€” it is the modern RDO/upstream default and *is* shipped in Epoxy. Rejected for Phase 2 because adopting it mid-build is a large by-hand rewrite: OVN replaces the L3/DHCP/metadata agents with `ovn-controller` + the OVN northbound/southbound databases, and tunnels with **Geneve, not VXLAN** (dropping the VXLAN/MTU-1450 specifics the plan is built on), and is harder to stand up and debug by hand than OVS. OVS preserves decision #14 and the agent-by-agent learning model with a one-driver change. OVN is the natural backend for the **Phase 3 Kolla rebuild**, where it is deployed for us. |
