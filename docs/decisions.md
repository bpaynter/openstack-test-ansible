# Decisions

Decisions made during planning and execution, with the reasoning behind each.

## Decisions taken

| # | Decision | Choice | Reason |
|---|---|---|---|
| 1 | Number of machines | **4** | Matches the 4 available 1G ports exactly; a 4th machine adds real value (3 OSD hosts + multiple compute nodes) over a 3-machine layout. |
| 2 | Machine to retire | **5080** (dead PSU) | Originally the 7050 was to be retired (weakest CPU), but the 5080's power supply failed and the 7050's PSU is incompatible, so the **5080** is retired as the donor instead and the **7050 returns** as the fourth node. The 7050's weak CPU is acceptable because it serves mainly as an I/O-bound OSD/compute node. |
| 3 | Controller | **7071** | 8 cores, mini-tower chassis, the most RAM (32GB), and now NVMe-booted. The natural "brain" of the cluster. |
| 4 | RAM split | **32 / 16 / 16 / 16** GB | The controller is the most service-dense node (control plane + Ceph MON/MGR), so it gets 32GB. Uses all 10 sticks (4+2+2+2), now across 7071/7060/5090/7050. |
| 5 | Ceph OSD layout | **5 OSDs across 3 hosts (2 + 1 + 2)** | Minimum 3 OSD hosts for default 3× replication with a `host` failure domain. 7060 carries 2 OSDs, 5090 carries 1, 7050 carries 2. Better balanced than the original 3+1+1, and all OSDs are matched 250GB SATA SSDs. (Per-tower SATA capacity is 3 *total* = 1 boot + 2 OSD, which caps each tower at 2 OSDs.) |
| 6 | Network speed | **1G, flat** | 10/100 is too slow for Ceph replication; a single flat network is fine for a learning cluster. Ceph saturating the 1G link is an intended, instructive observable. |
| 7 | Deployment method | **Phase 0 (prep/OS) → Phase 1 (manual Ceph + controller) → Phase 2 (hand-rolled Ansible compute) → Phase 3 (Kolla-Ansible rebuild)** | Explicit phased learning progression; each phase is motivated by the friction of the previous one. |
| 8 | Cluster lifespan | **Temporary (a few days)** | Build, benchmark, tear down. This is a test/learning cluster, not a permanent deployment. |
| 9 | Operating system | **AlmaLinux 9**, Minimal Install, SELinux enforcing | Switched from an initial lean toward AlmaLinux 10: RDO's EL10 packaging was still maturing in early 2026, which would bite the manual Phase 1 install hard. EL9 has mature, well-trodden RDO repos. |
| 10 | OpenStack release | **2025.1 "Epoxy"** | Recent release with complete, stable RDO/EL9 repos and lots of install-guide coverage — the sweet spot on EL9. |
| 11 | Ceph version | **Squid (19.x)** (Reef 18.x acceptable fallback) | Pairs with Epoxy-era OpenStack; installed via `centos-release-ceph-squid` on EL9. |
| 12 | Ceph bootstrap method | **cephadm** | Chosen by the user over the assistant's lean toward a fully-manual MON/MGR/OSD bring-up. Trade-off on record: cephadm is itself a container-orchestration layer, so Kolla in Phase 3 becomes the *second* such layer, not the first. Learning value holds because daemons are still placed by hand. |
| 13 | IP addressing | **Static IPs, no DHCP, no reservations** | Deterministic, always-known addressing (Ceph MONs configured by IP, Keystone endpoints are URLs, Kolla inventory is IP-based). Avoids dependence on the home router's lease table and guarantees no interference with the home network. Confirmed on subnet `192.168.1.0/24`: `controller` .130 (7071), `compute1` .131 (7060), `compute2` .132 (5090), `compute3` .133 (7050). (Name resolution and gateway settled separately — see #21.) Full address plan in [inventory.md](inventory.md). |
| 14 | Tenant networking | **VXLAN self-service** (Linux bridge driver) | Tenant networks tunnel VM traffic over the `192.168.1.0/24` underlay, sealing VM DHCP inside the overlay so it cannot race the home router's DHCP server (the "dual-DHCP" problem). A Neutron router NATs to a **flat provider/external network** on `192.168.1.0/24`; VMs reach the outside via **floating IPs**. Achieves the isolation a VLAN would give but in software, with no managed switch. (Supersedes the earlier "overlays + provider-on-its-own-VLAN" sketch.) |
| 15 | Ceph MON quorum | **4 MONs (controller + compute1/2/3), as auto-placed by cephadm** | Originally recorded as a deliberate **single** MON on the controller (accepted SPOF). In practice cephadm's default MON placement deploys a MON on each host as it is enrolled, so the live cluster actually runs **4 MONs** — confirmed by `ceph -s` (`1/4 mons down, quorum controller,compute1,compute2` during the compute3 outage). Resilience-positive: the intended single-MON SPOF never actually existed, so this is left as-is. Learning note: 4 is an *even* count — quorum is 3, so it tolerates only **one** MON failure (same as 3 would) and a 2/2 partition has no quorum; an odd count (3 or 5) is strictly better. Not worth pruning on a throwaway cluster, and Phase 3's Kolla sets its own MON count. |
| 16 | OSD memory target | **Lowered to ~1.5–2GB** (from the 4GB default) | The 16GB OSD nodes cannot afford 4GB BlueStore cache per OSD; baked into the Ceph config from the start. |
| 17 | Domain / DNS suffix | **`lab.internal`** | Confirmed by the user (FQDNs `controller.lab.internal`, `compute1–3.lab.internal`). Supersedes the earlier *proposed* `cluster.lab.internal`. A reserved/non-routable TLD that cannot collide with a real domain. |
| 18 | NVMe placement | **Both NVMe used as boot disks: 7071 (harvested from 5080) + 5090 (its own)** | The controller is the one node where root-disk latency materially helps (MariaDB, RabbitMQ, Keystone token churn touched by every benchmarked API call). Keeping the NVMe out of the OSD pool keeps all OSDs on matched SATA SSDs, avoiding a heterogeneous pool that would muddy benchmarks. |
| 19 | Per-tower SATA capacity | **3 SATA SSDs total per tower = 1 boot + 2 OSD** | The 7060 and 7050 each have 3 SATA bays/cables *total*, not 3 OSDs plus a boot disk. This caps each tower at 2 OSDs and drives the 2+1+2 topology. (Corrects an earlier over-count in the chunk-01 build plan.) |
| 20 | Hyperthreading | **Left enabled** on the compute nodes (7060, 5090, 7050) | RAM, not CPU, is the binding constraint on the 16GB nodes, so more logical CPUs is the more useful default; it is also the BIOS default (one less variable across four machines). The 7071 (i7-9700) has no HT. Caveat for benchmarking: HT adds variance to CPU-bound results — report vCPU counts as physical cores and interpret accordingly rather than toggling BIOS mid-project. |
| 21 | Name resolution | **Local `/etc/hosts` on all nodes, no DNS** | A 4-node throwaway cluster doesn't need real DNS; identical `/etc/hosts` files map all four FQDNs/short names. Nodes still use the normal LAN gateway for outbound internet (package/container pulls). |
| 22 | Firewall | **firewalld disabled** | Pragmatic for a short-lived, isolated lab — removes a whole category of "why can't this daemon talk" debugging. Keeping it on would mean opening a long per-service port list with little Phase 1 learning value. |
| 23 | Ceph hostnames | **FQDNs (`*.lab.internal`) with `cephadm bootstrap --allow-fqdn-hostname`** | Keeps Ceph identifiers consistent with the OS hostnames and `/etc/hosts`; the bootstrap check guards against mixing short and FQDN names, not against FQDNs. All `ceph orch host add` use FQDNs. Cost is purely cosmetic (longer daemon/host names). |
| 24 | Neutron mechanism driver | **Linux bridge** | Fewer moving parts to debug than OVS for a learning cluster; supports the VXLAN overlay (`enable_vxlan`, per-host `local_ip` tunnel endpoint). |
| 25 | Network node | **The controller (7071)** | The 7071 runs the Neutron L3, DHCP, and metadata agents in addition to its control-plane and Ceph MON/MGR roles. Concentrating the network node on the controller fits the single-NIC, days-long cluster. |
| 26 | Ansible scope | **Only the repeated compute-node work is rolled into Ansible** (`nova_compute`/`neutron_compute`, plus a throwaway `common` role for learning); controller-side Nova/Neutron is done **manually** (one-time) | The Ansible "seam" is repetition, and the one-time controller bring-up isn't repeated — forcing it into a role would add complexity without the payoff. Compute1/2/3 are the "same steps three times," which is where roles + per-host inventory variables earn their place. Also serves the learning goal: build understanding on a low-stakes `common` role first. |
| 27 | Ansible distribution | **Installed via `uv` (`uv tool install ansible --python 3.12`) — community 13.7 / `ansible-core` 2.20 on Python 3.12** | The AlmaLinux `dnf` package ships an end-of-life `ansible-core` (2.14/2.15) too old to match the live docs site (oldest published is community 11 / core 2.18). `uv` keeps it isolated from the system Python; pinning Python 3.12 was required because AlmaLinux's system Python 3.9 silently capped the resolver at `ansible` 8.7.0. Community 13 matches the docs "13" link exactly. The full community package was chosen (over bare `ansible-core`) so `openstack.cloud` is bundled for Stage 5. |
| 28 | Ansible privilege model | **Run as the normal login user; per-task `become` (default-off in `ansible.cfg`); `sudo` left password-protected (not passwordless); `-K` per escalating run** | Never run Ansible as root or via `sudo ansible-playbook` (that would SSH root→root and read root's keys/config). Default-off escalation keeps the privilege surface visible in the roles and makes a forgotten `become:` fail loudly rather than silently run as root (the inverse of the Phase 1 issue #3 root-owned-file class). Password-protected sudo is the deliberate security choice over passwordless; `-K` keeps the prompt explicit per run. |
| 29 | Per-host address variable | **One per-host `local_ip` (defined on all four nodes) serves as both the underlay IP and the VXLAN tunnel endpoint — no separate `underlay_ip`** | With one NIC per node, the host's `192.168.1.x` underlay address and its VTEP address are always the same value, so a second variable would be pure redundancy. `local_ip` is defined on the controller too (`.130`), since the controller is also a VTEP (it runs the L3/DHCP agents on tenant networks). The same variable is reused for inventory-driven `/etc/hosts` rendering. |
| 30 | cephadm SSH access | **Restore root SSH for cephadm, hardened: key-only (`PermitRootLogin prohibit-password`) + `from=`-restricted to the cluster nodes (`192.168.1.128/29`)** | cephadm manages every host over its **own** SSH channel as `root`, with a cluster keypair it generates — a separate control plane from Ansible. Root's authorized key had been pruned on the assumption that Ansible's login-user + `become` model (#28) made root SSH unnecessary; true for Ansible, but it broke cephadm (`CEPHADM_HOST_CHECK_FAILED` / ssh auth failure for root). cephadm needs root-level command execution, and the only two ways to give it that are root SSH or a non-root user with **passwordless** sudo (it is fully automated, so no `-K`-style prompt is possible). Chose to restore the root key — hardened to key-only via an sshd drop-in and `from=`-restricted so it only works from a cluster node (covers an mgr failover to any host; `192.168.1.128/29` spans `.128–.135`, i.e. all four nodes) — rather than add a NOPASSWD service user (see R10). Deliberately distinct from #28's Ansible login user: the two management planes keep separate, intentional privilege postures. |

## Decisions deliberately NOT taken (rejected options)

| # | Option considered | Verdict | Reason |
|---|---|---|---|
| R1 | Frankenstein power (110V→SATA adapters) for a 2nd SATA SSD in the SFFs | **Rejected** | External power bricks dangling from a sealed SFF chassis is a corruption risk that works for a week then quietly fails. A learning cluster does not need balanced OSDs, and the "really good reason" bar isn't met. |
| R2 | Drop the 7060 to free RAM (3-node, 4/4/2 sticks) | **Rejected** | Would cut Ceph to 2 OSD hosts — cannot satisfy 3× replication with a `host` failure domain — and would lose a 6-core compute node. The RAM gain (one node from 16→32GB) doesn't justify crippling the storage layer. |
| R3 | Use the 10/100 switch instead of 1G ports | **Rejected** | Ceph recovery over 100Mbit would be painful and teach bad latency lessons. Stay on 1G. |
| R4 | Kolla-Ansible-only build (skip the manual + hand-rolled Ansible phases) | **Rejected** | Although tempting given the temporary cluster, learning Ansible and OpenStack internals is an explicit goal, so the 3-phase progression was kept. |
| R5 | Drop to a 3-node cluster after the 5080 PSU failure | **Rejected** | A 3-node cluster (7071 + 7060 + 5090) would leave only 2 OSD hosts (7060 + 5090), which cannot satisfy 3× replication with a `host` failure domain. The weak 7050 is worth keeping precisely because it restores a third OSD host. |
| R6 | Put the harvested NVMe in a tower as an OSD (to reach 6 OSDs) | **Rejected** | One NVMe OSD in an otherwise all-SATA pool creates a heterogeneous pool that muddies benchmark numbers, and it would leave the controller booting from SATA. Matched OSDs + a fast NVMe controller is the better setup. |
| R7 | Boot the towers from USB / ODD bay to free all 3 SATA bays as OSDs | **Rejected** | USB boot disks are slow and flaky — an unwanted variable in a benchmark cluster. |
| R8 | Flat provider networking for tenants (VMs directly on `192.168.1.0/24`) | **Rejected** | Puts VM DHCP on the same L2 broadcast domain as the home router's DHCP server — a dual-DHCP race where either server can answer the other's clients. Disjoint pools prevent duplicate IPs but not the cross-answering. |
| R9 | VLAN-based tenant isolation | **Rejected** | Would cleanly solve the dual-DHCP race, but requires a managed switch that is not in play (the 10/100 switch was set aside). VXLAN achieves the same isolation in software. |
| R10 | Non-root cephadm SSH user with passwordless (NOPASSWD) sudo | **Rejected** | The documented way to avoid root SSH, but the user must have **passwordless** sudo (cephadm is fully automated — no prompt), which makes it root-equivalent in blast radius if its key is compromised, and adds a second privileged account to manage. For a throwaway lab the gain is policy/audit, not reduced privilege, so hardened root SSH (#30) was kept instead. |
